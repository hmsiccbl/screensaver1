v1.9.0, 2010.01.13

Small Molecule Data Sharing Levels ("DSL") [#1296]

  • Screen Data Sharing Levels
    • Screens can be assigned one of the following four data sharing levels, which affects what users can view the screen and what portions of the screen data are accessible to those users.
      Shared (Level 0)
      If the screen is a Small Molecule screen, any user with a "Small Molecule Screens" data sharing role (level 1, 2, or 3) can view the screen in its entirety. If the screen is an RNAi screen, any user with the "RNAi Screens" data sharing role can view the screen.
      Mutual Screens (Level 1)
      If the screen is a Small Molecule screen, all users with the "Small Molecule Screens Level 1" data sharing role can view the screen in its entirety; users with the "Small Molecule Screens Level 2" data sharing role can view the "mutual positives" data and screen summary information for screens that have overlapping positives with the user's own screens. If the screen is an RNAi screen, it is treated as Private (Level 3), below. [Note: Level 1 data sharing not yet implemented for RNAi screens].
      Mutual Positives (Level 2)
      If the screen is a Small Molecule screen, all users with the "Small Molecule Screens Level 1" or "Small Molecule Screens Level 2" data sharing role can view the "mutual positives" data and screen summary information for screens that have overlapping positives with the user's own screens. If the screen is an RNAi screen, it is treated as Private (Level 3), below. [Note: Level 2 data sharing not yet implemented for RNAi screens].
      Private (Level 3)
      Only the lab head, lead screener, and collaborators can view the screen
    • In Screen Viewer, a warning is issued to administrators when viewing a small molecule screen with a data sharing level that is too restrictive w.r.t. to the lab head's data sharing level
    • Well Search Results respects screen data sharing levels by controlling which Data Headers from other screen results are visible, as determined by the data access policy. When appropriate, only "positives" screen result data are made visible.
    • Users can view Lab Activities, CPRs, and Status Items of their own screens, but not those of others' screens
    • Screen Result "shareable" flag has been removed, having been functionally replaced by the new Screen data sharing level property.
  • User Data Access Roles
    • User roles have been partitioned into "Facility Usage Roles" and "Data Access Roles". The former are for administrative/reporting purposes, while the latter affect the types of data that a user can view.
    • The "Small Molecule User" data access role has been split into the following 3 roles (see above for how these are used):
      • Small Molecule User Level 1
      • Small Molecule User Level 2
      • Small Molecule User Level 3
    • Added "userRolesAdmin" role, to limit which administrators can edit users' data access roles. [#2042]

Miscellaneous Changes

  • User session inactivity timeout period has been increased to 120 minutes
  • Screen Result Importer optimizations, allowing the importer to operate in constant memory space. Screen Result import file sizes are no longer limited by available memory. [#2119]
  • Automatically update a screener's "Small Molecule Screener" and "RNAi Screener" facility usage roles when the user is associated with or dissociated from a screen. [#2040]
  • Added "Update History" sections to Screen Viewer and User Viewer. Currently, only used to track edit/save events, changes to data sharing levels for user and screens, changes to data access roles for users, and changes to lead screener and lab head for screens. [#1300]

Major Domain Model Changes (for developers)

  • Added AssayWell entity, to group ResultValues for a single well within a ScreenResult. [#1344], [#2067]
  • Added AuditedAbstractEntity superclass, which provides an audit log (change history) capability to entity subclasses (Study, Screen, ScreensaverUser, Activity, Library, CherryPickRequest). [#1300]
  • Screen and Screen Result "shareable" flags have been removed (note that only the Screen Result flag was exposed in the UI previously). Replaced by the new Screen "data sharing level" property.

User Interface Infrastructure (for developers)

User interface infrastructure rework for viewer/editor backing beans. The updated backing bean abstract class hierarchy (EntityViewer, et.al.) now implements viewer/editor lifecycle methods (edit, validate, save, cancel, and other internal hooks) in a consistent fashion, replacing the numerous varying implementations in viewer backing bean classes. The varying implementations caused difficulty during maintenance and debugging, as the developer could not easily reason about the session/transactional behavior without studying each entity viewer's custom implementation. In particular, many Hibernate-related problems have been addressed by standardizing the session/transactional design and ensuring that entity instances are all re-attached to the session at a single point in time (for edit/save operations). See rev 3676

v1.8.0, 2009.09.03

The following is a summary of the changes introduced by Screensaver version 1.8.0.  It is not comprehensive.  For more details:
  • See Tickets resolved in 1.8.0
  • Query Subversion repository /branches/iccbl/lib-mgmt-rework (commit logs and diffs) for a full description of the changes
  • Inquire on the screensaver-development mailing list

Library Contents Versions

  • The reagent information for a library can be reloaded, allowing the information to be updated and/or corrected, while keeping past versions of the library contents information intact for auditing purposes [#1765].
    • The Screensaver database becomes the master repository of library information, freeing the facility from having to store past library contents versions as external files.
    • As gene and compound information provided by third-party resources (NCBI, etc.) evolves, new library contents versions can be loaded to reflect more up-to-date information.
    • Vendor-provided corrections to library information can be loaded, while maintaining a record of the formerly erroneous information.
  • When a new version of a library's contents are imported by an administrator, the new version is initially viewable only by administrators, but after review can be "released", making it visible to normal users [#1907].
  • Library contents versions that have not yet been released may be deleted [#1908].

Library Contents Loading

  • New file formats for library contents loading:
    • RNAi Libraries
    • Small Molecule Libraries
  • Each library contents loading is tracked via an Administrative Activity, recording who/when/why for the loading activity.
  • Queries that were made to online services (NCBI, etc.) during library loading operation have been eliminated [#1362].  Data that were previously acquired via these network queries are now represented by explicit fields in the library contents input files.  This makes library loading a significantly faster operation, and more transparent in terms of what data is actually being loaded into the database.  Moved the respective web service clients to edu.harvard.med.iccbl.screensaver, as they are no longer part of Screensaver proper.  They are now only used outside of the running Screensaver application as a means of obtaining the data needed for library contents input files: at ICCB-L we have packaged these Java classes into Pipeline Pilot components, which we use to generate the library contents input files.
  • Removed dynamic calculation of molecular mass and formula [#1338].  These data are specified via explicit fields in the small molecule library contents input file and stored in the database.
  • Removed dynamic calculation of SMILES and InChi from molfile chemical structures [#1767].  These data are specified via explicit fields in the small molecule library contents input file and stored in the database.
  • Gene information provided by library vendors is stored independently from the gene information that is provided/updated by the facility [#1586].  Previously, some vendor-provided gene attributes were overwritten by Screensaver during the loading process if updated information was acquired via calls to NCBI.  Now, vendor-provided gene information can be stored verbatim, while a facility-provided gene information is stored independently (in general, the facility gene information will be derived from the vendor gene information, with updates made as desired).
  • For RNAi pool libraries, pool-to-duplex well deconvolutions are stored explicitly in the database and specified in the RNAi library contents input file [#1204].  Screensaver no longer dynamically calculates this mapping.
  • New LibraryContentsLoader command-line application.
  • Removed background loading of RNAi libraries, as it did not work well with respect to error reporting and was not supported for other library types [#1610].

User Interface

  • Library Viewer has new section, "Library Contents Versions" (administrators only).  Allows administrators to view, import, delete, and release contents versions for a library.
  • Well Viewer has new sections:
    • "Other Wells With Reagents" (used to be in Gene Viewer, which is now defunct)
    • "Duplexes", shows the duplexes for RNAi library pool wells
  • Now that reagent information (compounds, genes, silencing reagents, etc.) is managed by each well independently, the following pages have been eliminated:
    • Reagent Search Results (subsumed by Well Search Results).  The "Find Reagents" command shows results Wells Browser, potentially showing multiple well rows for a single reagent vendor identifier.
    • Gene Viewer (subsumed by Well Viewer)
    • Compound Viewer (subsumed by Well Viewer)
  • "Well Type" renamed to "Library Well Type", to differentiate from "Assay Well Type".
  • Facility gene information is exposed to end users, whereas vendor gene information is currently only accessible via direct database queries.
  • Reagent vendor IDs, previously shown as a single field "<vendor>:<ID>" now exposed as separate columns in data tables, for more effective sorting and filtering. [#1256]
  • Silencing Reagent sequences exposed by user interface, but only to administrators [#1209].

Scalability Improvements (to handle larger data set sizes)

  • Reduced memory usage during data table export operations (Excel and SDF exports)
  • Reduced memory usage during library contents loading operations
  • Export to Excel writes out to multiple worksheets, if necessary [#1990]

Miscellaneous

  • Small molecule image structures are now retrieved from pre-generated static files, by default, and are no longer dynamically generated [#1192].  This removes Screensaver's run-time dependency on Daylight Chemical software. 
  • Fixed Cherry Pick Request Viewer's "Record Failed Assay Plates" command [#2012].
  • Fixed error that occurred when user specified a funding support on new screen [#1950].
  • "Fixed" incorrect molecular mass and formula calculations by removing run-time calculations altogether, and now requiring values to be specified in library contents input file (now stored with small molecule reagents)[#1358], [#1554].
  • Added "miRNA" silencing reagent type [#1961].
  • cellHTS2 feature can enabled or disabled via system property in screensaver.properties [#840].

Domain Model (for developers)

  • Renamed "Compound" to "SmallMoleculeReagent" [#1270].
  • Added NaturalProductReagent [#1769].
  • Reagent entity is now parent of SmallMoleculeReagent, SilencingReagent, and NaturalProductReagent.  Each well may have a single reagent.
  • The entities representing reagent information are no longer shared among multiple wells.  Each well "owns" it own reagent records, and so there is no longer an attempt to maintain canonical versions of reagents or genes.  This includes compound information (was Compound, now SmallMoleculeReagent), Gene, and SilencingReagent.  This allows the contents of different libraries to be updated (by adding a new contents version) without affecting other libraries that may contain the same reagents. This also allows for an overall simplified design and library-loading performance improvements.
  • Explicit relationship from pool SilencingReagent to the wells containing the constituent duplex SilencingReagents.  This allows a pool well to maintain a single pool reagent, while allowing information about all duplexes to be acquired efficiently and accurately [#1204].
  • SmallMoleculeReagent now has molecular mass, molecular weight, and molecular formula fields [#1338, #1963].
  • Single reagent per well (either a SmallMoleculeReagent, SilencingReagent, or NaturalProductReagent).
    • Previously, a well could contain multiple compounds, though this only occurred for "composite" compounds (usually a compound and a salt).  However, since such "composite" small molecules can be handled by within a molfile structure, SMILES and InChi identifiers, and structure images, there is  no need to break them up into multiple SmallMoleculeReagents.  This also eliminates the complexities of determining/maintaining a "primary" compound.
    • Previously, a well could maintain multiple silencing reagents to handle pool wells.  Now, pools of silencing reagents are considered a single reagent, with explicit associations to to associated duplex silencing reagents [#1204].
    • Well tracks its historical set of "reagents", one reagent per library contents version. Only one of these reagents is active at a given point in time, represented by the "latestReleasedReagent" relationship.
  • Well entity changes:
    • molfile, smiles moved to SmallMoleculeReagent
    • genbankAccessionNumber moved to Gene.genbankAccessionNumbers (exclusively)
    • gene now in SilencingReagent
  • SilencingReagent now has two distinct gene records: vendor ("theirs"), facility ("ours").
  • Library stores pre-calculated experimental well count value in database, for performance [#1292].
  • Added Library.isPool flag [#1954].
  • Removed:
    • edu.harvard.med.screensaver.model.derivatives package and its entities [#1291]
    • Gene.oldEntrezegeneIds and Gene.oldEntrezegeneSymobls
    • Many Library attributes (see diff from 3020 to 3019, and 3224) [#1494]
    • CAS, NSC numbers for small molecules

Refactorings and APIs (for developers)

  • Library parsers/loaders now unified as subclasses of LibraryContentsParser, all invoked via LibraryContentsLoader.
  • RelationshipPath and PropertyPath [#1225] [rev 3102 commit log].  Supports building paths incrementally via method calls, rather than dot-separated strings, providing improved compile-time safety. Model entities can now statically define their single-hop relationships as RelationshipPaths, allowing compile-type checking of domain model structure.
  • TableColumn hierarchy [rev 3236 commit log].  Combined parallel hierarchies of TableColumn subclass and EntityColumn subclasses by factoring out RelationshipPath-related methods into HasFetchPatch interface.
  • Domain model automated unit testing framework [see rev 3120-3173 commit log].  Major rewrite.
  • edu.harvard.med.screensaver.io.Workbook2 [see rev 3174 commit log].  Support for iterator-style row and column reading.
  • Workbook-based library contents parsers now use jxl (JExcel API) instead of POI/HSSF library.

Build-related (for developers)

  • Removed CDK library [#1338].
  • Removed OpenBabel library [#1767].
  • Removed JSFUnit support from build.xml, as these JSF units tests are no longer being maintained.
  • edu.mit.* classes (SOAP) must now be compiled via chembank-wsimport Ant target, as it is no longer managed in source control repository.
  • Support for Cobertura unit test code coverage in build.xml.
  • Support for FindBugs in build.xml (fixed all high-priority warnings, and many others).

v1.5.0, 2008.02.11

The implementation of Search Result data tables has been entirely re-designed, providing the following benefits:

  • Improved performance and scalability for large data sets. In particular, sorting and filtering of large data sets is performed by the database, not by Screensaver. Near minimal sets of database operations are performed in response to user actions (open, scroll, sort, add/remove columns, etc.).
  • Supports search results composed of columns across related entity types (e.g., Library, Well, Gene, Silencing Reagent), with user-configurable column selections, via a hierarchical column selection interface.
  • For developers, simplified definition of Search Result pages, with nearly-declarative implementation (simply provide a name and an entity property binding for each column). Dynamically configurable data access strategy ("load all" or "virtual scrolling/paging"). No data access code required (i.e., no Hibernate queries).

Users should note:

  • Screen Viewer's "Data" panel now has a full search interface, replacing the old "Show All, Show Plate, Show Positives" interface.
  • Screen Viewer's "Data" panel now supports combining data columns from multiple screen results (data headers) and studies (annotations).
  • Well Search Results pages should load faster for large data sets. For example, when viewing all wells for a library (accessed from the Library Viewer page).
  • Creative use of the Search and Column selection features provide a basic level of custom reporting throughout Screensaver.
